# Subhuti AI Skill 设计规范
> 版本：V1.0  
> 适用架构：Subhuti 四层轻量 Agent 框架（记忆层 / 运行层 / 流程层 / 扩展层）

## 一、Skill 核心定义
**Skill 是 Agent 的专项能力包**，是面向业务场景的能力封装单元，粒度高于原子 Tool。
- **Tool**：原子函数，仅负责单一底层操作（如数据库查询、HTTP 请求、文件读写），无业务逻辑编排；
- **Skill**：场景化能力包，包含参数校验、多工具编排、结果处理、记忆交互、错误兜底，是 Agent 对外能力的直接载体。

设计目标：让 AI 清晰知道「什么时候用、怎么用、用完返回什么」，同时保持框架轻量、可插拔、无黑盒。

---

## 二、Skill 八大核心组成要素
所有要素严格对应 Subhuti 四层架构，职责单一、边界清晰。

| 要素层级 | 要素名称 | 核心作用 | 归属架构层 |
|----------|----------|----------|------------|
| 元数据层 | 身份与触发标识 | LLM 识别能力入口，决定何时调用 | 运行层 |
| 输入层 | 输入契约与校验 | 结构化定义入参，减少传参错误 | 运行层 · 约束模块 |
| 逻辑层 | 核心能力本体 | Skill 的具体执行逻辑与编排 | 运行层 + 流程层 |
| 依赖层 | 工具与资源声明 | 明确依赖的底层 Tool、外部资源 | 运行层 · 工具系统 |
| 约束层 | 执行硬规则 | 代码级兜底限制，不依赖模型自律 | 运行层 · 约束模块 |
| 输出层 | 输出规范 | 统一返回格式，稳定解析下游 | 流程层 · Observe 阶段 |
| 记忆层 | 记忆交互规则 | 明确对三层记忆的读写权限 | 记忆层交互接口 |
| 容错层 | 错误与兜底 | 失败降级、重试策略，提升稳定性 | 流程层 · Reflect 阶段 |

### 1. 元数据层：身份与触发标识
- **核心字段**
    - `name`：唯一标识，动词开头、蛇形命名，如 `search_long_memory`
    - `description`：最核心字段，公式 = 功能说明 + 适用场景 + 边界限制，直接决定 LLM 调用准确率
    - `tags`：分类标签，用于批量管理与路由
- **设计要点**：元数据常驻上下文，详细执行逻辑触发后加载，避免无效占用 Token，契合「按需检索」理念。

### 2. 输入契约：参数规范与校验
- 采用标准 JSON Schema 格式，声明字段类型、必填项、默认值、格式校验规则；
- 校验逻辑在代码层硬执行，不依赖 LLM 传参准确性。

### 3. 核心逻辑：能力本体
分为两类实现形态：
- **提示词驱动型**：步骤化指令、领域规则、评分标准，纯 LLM 推理，无代码执行；
- **代码执行型**：工具调用链编排、业务逻辑处理、数据转换，确定性高。

### 4. 工具依赖：底层能力声明
- Skill 不直接实现底层能力，只做编排；
- 明确声明依赖的所有原子 Tool、外部服务、资源文件，底层 Tool 可独立替换，不影响 Skill 逻辑。

### 5. 执行约束：硬规则兜底
- 最大执行次数、超时时间、权限要求、限流规则；
- 禁止操作边界，从代码层面拦截高危行为。

### 6. 输出规范：结果格式标准化
- 统一成功/失败状态标识；
- 明确返回字段说明、格式要求、长度限制与摘要规则；
- 输出结果结构化，方便 LLM 解析与下游流程处理。

### 7. 记忆交互：与记忆迷宫的联动规则
明确 Skill 对三层记忆的读写权限：
- 读取：是否允许读取短期/长期/知识库记忆；
- 写入：是否将执行结果归档到长期记忆、是否更新知识库；
- 检索：是否主动触发记忆检索。

### 8. 错误与兜底：失败降级策略
- 重试次数、重试间隔规则；
- 失败降级返回提示；
- 错误信息格式化规范。

## 三、两种主流 Skill 形态
### 1. 提示词驱动型（知识/流程类）
- **本质**：结构化 Prompt 工程，将领域知识、操作步骤封装为可复用能力包；
- **标准结构**：`SKILL.md`（YAML 头元数据 + Markdown 正文指令）+ 参考资料；
- **典型示例**：简历优化、代码规范评审、需求文档拆解；
- **特点**：灵活度高，无代码执行，纯 LLM 推理。

### 2. 代码执行型（工具/操作类）
- **本质**：多工具编排 + 业务逻辑封装，结果确定性强；
- **标准结构**：元数据 + 入参校验 + 工具调用链 + 结果处理；
- **典型示例**：长期记忆检索、数据库查询、文件格式转换；
- **特点**：结果可控，适合固定流程，是记忆迷宫的核心载体。

## 四、标准编写步骤（从 0 到 1）
1. **定义能力边界**：单一职责，明确「能做什么、绝对不能做什么」，拒绝大而全的万能 Skill；
2. **编写触发描述**：遵循「功能 + 场景 + 边界」公式，保障 LLM 识别准确率；
3. **设计输入输出契约**：像定义后端接口一样规范 Schema，不依赖模型猜参数；
4. **编写核心逻辑**：步骤化指令或工具调用编排，逻辑线性清晰；
5. **补充约束与兜底**：硬规则写入代码，覆盖错误场景降级方案；
6. **声明记忆交互规则**：明确对三层记忆的读写权限；
7. **测试调优**：验证触发准确率、参数合规性、结果符合度，迭代优化描述与逻辑。

## 五、Subhuti 框架落地实现
### 1. 核心 Trait 定义
```rust
use async_trait::async_trait;
use serde_json::Value;
use crate::memory::MemoryStore;
use crate::runtime::tool::Tool;
use crate::runtime::constraint::ConstraintRules;

/// Skill 统一特征：面向场景的完整能力包
#[async_trait]
pub trait Skill: Send + Sync {
    // ========== 元数据 ==========
    fn name(&self) -> &str;
    fn description(&self) -> &str;
    fn tags(&self) -> Vec<&str>;

    // ========== 输入契约 ==========
    fn parameters_schema(&self) -> Value;

    // ========== 核心执行 ==========
    async fn execute(&self, args: Value, ctx: &SkillContext<'_>) -> SkillResult;
}

/// Skill 执行上下文：依赖注入，Skill 本身无状态
pub struct SkillContext<'a> {
    pub memory: &'a dyn MemoryStore,
    pub tools: &'a [Box<dyn Tool>],
    pub constraints: &'a ConstraintRules,
}

/// Skill 统一返回结构
pub enum SkillResult {
    Success(String),
    Failed(String),
}
```

### 2. 完整示例：长期记忆检索 Skill
```rust
pub struct SearchLongMemorySkill;

#[async_trait]
impl Skill for SearchLongMemorySkill {
fn name(&self) -> &str {
"search_long_memory"
}

    fn description(&self) -> &str {
        "检索历史归档的长期对话记忆。当用户询问之前聊过的内容、且当前会话短期记忆中没有相关信息时调用。仅支持关键词精准检索，不支持语义模糊匹配。"
    }

    fn tags(&self) -> Vec<&str> {
        vec!["memory", "search"]
    }

    fn parameters_schema(&self) -> Value {
        serde_json::json!({
            "type": "object",
            "properties": {
                "keyword": {
                    "type": "string",
                    "description": "检索核心关键词，尽量简短精准"
                }
            },
            "required": ["keyword"]
        })
    }

    async fn execute(&self, args: Value, ctx: &SkillContext<'_>) -> SkillResult {
        // 1. 参数前置校验
        let keyword = match args["keyword"].as_str() {
            Some(k) => k,
            None => return SkillResult::Failed("缺少必填参数 keyword".into()),
        };

        // 2. 调用记忆层检索（通过抽象接口，不耦合具体存储实现）
        let results = ctx.memory.search_long(keyword).await;

        // 3. 结果格式化，便于 LLM 解析
        if results.is_empty() {
            SkillResult::Success("未检索到匹配的长期记忆".into())
        } else {
            let formatted = results
                .iter()
                .enumerate()
                .map(|(i, text)| format!("{}. {}", i + 1, text))
                .collect::<Vec<_>>()
                .join("\n");
            SkillResult::Success(format!("检索到相关长期记忆：\n{}", formatted))
        }
    }
}
```

### 3. 架构对应关系
- 运行层：Skill 定义与注册，无状态纯能力单元；
- 流程层：ReAct 循环统一调度 Skill，与调度 Tool 逻辑一致；
- 记忆层：Skill 通过上下文注入访问记忆，不直接操作底层存储，实现解耦；
- 扩展层：可通过 Hook 拦截 Skill 执行前后，扩展日志、缓存、权限校验能力。
## 六、核心设计原则
- 单一职责：一个 Skill 只解决一类问题，避免万能 Skill 导致 LLM 调用混乱；
- 描述优先：description 质量直接决定调用准确率，优先级高于实现逻辑；
- 无状态设计：Skill 本身不存储数据，所有依赖通过上下文注入，支持水平扩展；
- 渐进披露：元数据常驻上下文，详细逻辑触发后加载，节约 Token 开销；
- 硬约束兜底：关键限制写入代码层，不依赖 LLM 自觉遵守；
- 记忆解耦：Skill 不直接操作存储底层，通过记忆层抽象交互，存储实现可无缝替换。
