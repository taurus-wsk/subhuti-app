## 设计哲学

### 核心理念：薄封装 + 强心灵 + 插件化

```
┌─────────────────────────────────────────────────────────┐
│                    设计理念三角                           │
│                                                          │
│     薄封装（主框架精简）    ←→    强心灵（动态演化）      │
│                    \                                      │
│                     \                                     │
│                      →  插件化（专家生态）  ←            │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### 三大原则

| 原则 | 说明 | 体现 |
|------|------|------|
| **薄封装** | 主框架只保留核心能力，不做重封装 | LLM、记忆、运行时 |
| **强心灵** | Agent 有"灵魂"，会成长、有性格 | 心灵层 + 心灵宫殿 |
| **插件化** | 领域能力通过专家插件扩展 | ExpertPlugin trait |

### 为什么这样设计？

1. **成本效率**：单 Agent 强化 > 多 Agent 协作，降低调用成本
2. **个性化**：每个 Agent 都有独特性格，越用越懂你
3. **扩展性**：领域知识通过插件注入，不污染主框架
4. **可观测**：完整的 Trace 和健康检查系统

### Trace 追踪链

每个请求都有完整的追踪链路：

```

  ├─ ExpertDetectionSpan    (专家检测)
```

### 插件钩子机制

| 钩子 | 触发时机 | 用途 |
|------|----------|------|
| `on_load` | 插件加载时 | 初始化资源 |
| `on_activate` | 插件激活时 | 注入人格、加载知识库 |
| `before_message` | 消息处理前 | 修改输入、预处理 |
| `after_message` | 消息处理后 | 修改输出、后处理 |
| `on_deactivate` | 插件停用时 | 清理资源 |
| `on_unload` | 插件卸载时 | 释放资源 |

## 关键技术决策

### 1. 为什么用 Rust？

- **性能**：内存安全、零成本抽象，适合高并发 Agent
- **可靠性**：类型系统减少运行时错误
- **生态**：Tokio 异步运行时、Axum Web 框架

### 2. 为什么心灵层在 Skill 层之上？

- 人格影响所有技能的表现
- 记忆是全局共享的资源
- 角色养成是框架核心特色

### 3. 为什么用插件而不是多 Agent？

- **成本**：单 Agent + 插件 < 多 Agent 协作
- **一致性**：统一人格，不会出现角色分裂
- **简单性**：插件管理比 Agent 编排简单
