## 标准 Agent 框架对比：可能缺失的模块
### 🔴 高优先级（核心能力） 1. 提示词管理模块 (Prompt Management)
- 现状 ：提示词散落在各个 Skill、Flow、Expert 中，硬编码在字符串里
- 缺失 ：统一的提示词模板管理、版本管理、变量注入、A/B 测试
- 建议 ：创建 prompt 模块，支持模板化、多语言、动态组装 2. 缓存系统 (Caching)
- 现状 ：LLM 调用每次都走真实 API，没有缓存
- 缺失 ：LLM 响应缓存（语义缓存）、工具结果缓存、Embedding 缓存
- 建议 ：创建 cache 模块，支持精确匹配 + 语义相似度缓存 3. 评估/评测模块 (Evaluation)
- 现状 ：只能人工测试，没有自动化评测
- 缺失 ：评测数据集管理、自动评分、回归测试、质量监控
- 建议 ：创建 eval 模块，支持用例集、多维度评分、对比评测
### 🟡 中优先级（生产就绪） 4. 限流与配额 (Rate Limiting)
- 现状 ：没有调用频率限制
- 缺失 ：按用户/按 IP/按 API Key 的限流、Token 配额管理、费用统计
- 建议 ：创建 rate_limit 模块，或集成到 runtime 中 5. 任务队列 / 长时任务 (Task Queue)
- 现状 ：所有任务都是同步 HTTP 请求处理
- 缺失 ：长时运行任务、后台任务、任务状态跟踪、取消/重试
- 建议 ：创建 task 模块，支持异步任务队列 6. 工具沙箱 (Tool Sandbox)
- 现状 ：工具直接执行，没有安全隔离
- 缺失 ：代码执行沙箱、文件系统隔离、权限控制、资源限制
- 建议 ：在 runtime/tools 中增加沙箱层，支持 Docker/WASM 隔离 7. 多模态支持 (Multimodal)
- 现状 ：纯文本交互
- 缺失 ：图片理解、语音输入输出、文件处理（PDF/Excel 等）
- 建议 ：在 runtime 中增加 multimodal 子模块
### 🟢 低优先级（增强体验） 8. 配置中心 (Config Center)
- 现状 ：配置从文件加载，启动后固定
- 缺失 ：动态配置更新、热加载、按环境区分、配置版本管理
- 建议 ：增强现有 config，支持运行时动态调整 9. 向量数据库抽象 (Vector DB Abstraction)
- 现状 ：embedding 直接用 Ollama，向量存储在 PostgreSQL
- 缺失 ：多种向量数据库支持（Milvus/Qdrant/Weaviate）、统一接口
- 建议 ：在 memory 中抽象向量存储层 10. Agent 模板市场 (Agent Templates)
